# Material-UI Docs Code Examples Guide

This document explains how code examples (demos) work in the Material-UI documentation system, including how they're defined, processed, rendered, and how multiple files and previews are handled.

## Overview

The Material-UI docs system supports interactive code examples that can:
- Display live, editable React components
- Support multiple files (relative imports)
- Show TypeScript and JavaScript variants
- Generate preview snippets for quick viewing
- Be embedded in markdown documentation

## How Examples Are Defined

### In Markdown Files

Examples are referenced in markdown files using a special syntax:

```markdown
{{"demo": "path/to/Example.tsx"}}
```

Or with additional options:

```markdown
{{"demo": "path/to/Example.tsx", "hideToolbar": true, "defaultCodeOpen": true}}
```

The demo path is relative to the markdown file's location in the `docs/data` directory structure.

### Example File Structure

For a demo at `docs/data/material/components/button/BasicButton.tsx`, you would reference it as:

```markdown
{{"demo": "components/button/BasicButton.tsx"}}
```

### Demo Options

- `hideToolbar`: Hides the demo toolbar (for non-interactive examples)
- `defaultCodeOpen`: Opens the code view by default
- `bg`: Background style (`'outlined'`, `'gradient'`, `'playground'`, or `true`)
- `maxWidth`: Maximum width of the demo
- `height`: Height of the demo
- `iframe`: Render in an iframe (for isolated demos)
- `isolated`: Isolate the demo in a sandbox
- `disableLiveEdit`: Disable live code editing
- `aiSuggestion`: Enable AI suggestions for the demo

## File Structure and Processing

### Main Demo Files

Each demo consists of:

1. **TypeScript file** (`.tsx`): The primary source file
   - Example: `SessionsChart.tsx`

2. **JavaScript file** (`.js`): Auto-generated from TypeScript
   - Generated by `docs/scripts/formattedTSDemos.js`
   - Example: `SessionsChart.js`

3. **Preview file** (`.tsx.preview`): Optional preview snippet
   - Auto-generated by Babel plugin during TypeScript transpilation
   - Contains a simplified version of the JSX (max 16 lines)
   - Example: `SessionsChart.tsx.preview`

### Multiple Files Support

Demos can import relative modules, which are automatically detected and included:

```typescript
// Main demo file
import { data } from './data';
import { helper } from './utils/helper';

export default function MyDemo() {
  // ...
}
```

The system automatically:
1. Detects relative imports (starting with `./` or `../`)
2. Resolves file extensions (`.js`, `.jsx`, `.ts`, `.tsx`)
3. Includes these files as separate tabs in the code viewer
4. Supports nested relative imports recursively

### Chart Example: SessionsChart

A real-world example using multiple files can be found in:

- **Location**: `docs/data/material/getting-started/templates/dashboard/components/SessionsChart.tsx`
- **Features**:
  - Uses `@mui/x-charts/LineChart` component
  - Includes helper functions (`AreaGradient`, `getDaysInMonth`)
  - Demonstrates theme integration
  - Shows complex data visualization

The chart example demonstrates:
- TypeScript with proper types
- Theme integration via `useTheme()`
- Complex component composition
- Multiple data series rendering

## Processing Pipeline

### 1. Markdown Parsing

The markdown loader (`packages/markdown/loader.mjs`) processes markdown files:

1. **Extracts demo references**: Finds all `{{"demo": "..."}}` patterns
2. **Loads demo files**: Reads the actual `.tsx` and `.js` files
3. **Detects relative imports**: Scans for relative imports in demo files
4. **Loads preview files**: Reads `.tsx.preview` files if they exist
5. **Builds demo object**: Creates a structured object with all code variants

### 2. Preview Generation

Previews are generated during TypeScript transpilation:

**Process**:
1. Babel plugin (`docs/src/modules/utils/babel-plugin-jsx-preview.js`) analyzes the AST
2. Extracts the return statement's JSX
3. Removes wrapper components (`div`, `Box`, `Stack`) if they only contain children
4. De-indents the code
5. Writes to `.tsx.preview` if the result is ≤ 16 lines

**Preview File Example**:
```tsx
<ToggleButtonGroup
  color="primary"
  value={alignment}
  exclusive
  onChange={handleChange}
  aria-label="Platform"
>
  <ToggleButton value="web">Web</ToggleButton>
  <ToggleButton value="android">Android</ToggleButton>
  <ToggleButton value="ios">iOS</ToggleButton>
</ToggleButtonGroup>
```

### 3. Demo Loading

The webpack loader (`packages/markdown/loader.mjs`) creates:

```javascript
export const demos = {
  "Example.tsx": {
    module: "./Example.js",
    moduleTS: "./Example.tsx",
    raw: "/* JS code */",
    rawTS: "/* TS code */",
    jsxPreview: "/* preview snippet */",
    relativeModules: {
      JS: [{ module: "./data.js", raw: "..." }],
      TS: [{ module: "./data.ts", raw: "..." }]
    }
  }
};
```

## Rendering System

### Component Hierarchy

```
MarkdownDocs
  └─ RichMarkdownElement (for each content block)
      └─ Demo (for demo entries)
          ├─ DemoSandbox (wraps the component)
          ├─ DemoToolbar (controls)
          └─ Code Viewer/Editor
```

### Demo Component (`docs/src/modules/components/Demo.js`)

The main demo component handles:

1. **Code Variants**: Switches between JS/TS based on user preference
2. **Live Editing**: Uses `ReactRunner` for live code execution
3. **Preview Mode**: Shows preview snippet when code is closed
4. **Multiple Files**: Displays tabs for relative modules
5. **Copy Functionality**: Allows copying code to clipboard

### Demo Toolbar Controls

The demo toolbar (`docs/src/modules/components/DemoToolbar.js`) provides a comprehensive set of controls for interacting with demos. All controls are accessible via keyboard navigation (arrow keys, Home, End).

#### Primary Controls (Left Side)

1. **Open MUI Chat** (`OpenMuiChat` component)
   - Opens AI chat with the demo code pre-loaded
   - Allows asking questions about the demo
   - Positioned on the far left of the toolbar

2. **JS/TS Toggle Buttons** (only visible when code is open)
   - Toggle between JavaScript and TypeScript code variants
   - JS button: Shows JavaScript version of the code
   - TS button: Shows TypeScript version (disabled if TS variant not available)
   - Uses `ToggleButtonGroup` for exclusive selection
   - Hidden when code panel is closed

#### Primary Controls (Right Side)

3. **Show/Hide Source Button**
   - Toggles the code panel open/closed
   - Label changes based on state:
     - "Show source" / "Show full source" (when closed)
     - "Hide source" / "Hide full source" (when open)
   - When showing preview, label indicates "full source"

4. **StackBlitz Button** (icon button)
   - Opens the demo in StackBlitz
   - Creates a new StackBlitz project with the demo code
   - Only shown if `hideEditButton` option is not set

5. **CodeSandbox Button** (icon button)
   - Opens the demo in CodeSandbox
   - Creates a new CodeSandbox with the demo code
   - Only shown if `hideEditButton` option is not set

6. **Copy Source Button** (icon button)
   - Copies the current code to clipboard
   - Icon changes to checkmark when copied
   - Copies the active tab's code (if multiple files)
   - Shows visual feedback via icon change

7. **Reset Focus Button** (icon button)
   - Resets keyboard focus to the initial focus point
   - Helps with accessibility navigation
   - Uses `CenterFocusWeak` icon

8. **Reset Demo Button** (icon button)
   - Resets the demo to its original state
   - Clears any code edits made in the editor
   - Restores the initial code
   - Uses `RefreshRounded` icon

9. **More Menu** (three dots icon)
   - Opens a dropdown menu with additional options:
     - **View on GitHub**: Opens the demo file on GitHub
     - **Copy source link (JS)**: Copies a direct link to the JS version
     - **Copy source link (TS)**: Copies a direct link to the TS version
     - **Dev menu items** (staging/PR environments only):
       - Link to demo on PR preview
       - Link to demo on `next` branch
       - Demo permalink
       - Link to demo on `master` branch

#### File Tabs (Below Toolbar)

When a demo has multiple files (relative imports), tabs are displayed:

- **Tab Navigation**: Shows all files as tabs
- **Active Tab**: Displays the code for the selected file
- **Tab Names**: Shows the relative file path (e.g., `./data.ts`, `./utils/helper.ts`)
- **Live Editing**: Only the main file (first tab) supports live editing
- **Other Tabs**: Show read-only code view

#### Toolbar Behavior

- **Visibility**: Toolbar is hidden on mobile (`<sm` breakpoint)
- **Keyboard Navigation**: 
  - Arrow keys navigate between controls
  - Home/End jump to first/last control
  - Tab order is managed programmatically
- **Focus Management**: Controls can be hidden/shown based on code panel state
- **Accessibility**: All controls have proper ARIA labels and roles

### Rendering Modes

1. **Bundled Component**: Pre-rendered component (faster, no live editing)
2. **Live Component**: Rendered via `ReactRunner` (supports live editing)
3. **Preview Mode**: Shows preview snippet when code panel is closed

### Editable Code System

The editable code feature allows users to modify code and see live updates. Here's how it works:

#### Code Editor Component

The `DemoEditor` component (`docs/src/modules/components/DemoEditor.tsx`) provides the editable interface:

1. **Editor Library**: Uses `react-simple-code-editor` for the text editing interface
2. **Syntax Highlighting**: Uses Prism.js via `@mui/internal-markdown/prism` to highlight code
3. **Language Support**: Supports both JavaScript (`jsx`) and TypeScript (`tsx`) syntax highlighting
4. **Real-time Updates**: As users type, the code is passed to `ReactRunner` for live execution

**Key Features**:
- Syntax-highlighted code editing
- Keyboard shortcuts (Enter to focus, Escape to blur)
- Copy button integration
- Error display integration

#### TSX/JSX Live Rendering

The live rendering is accomplished through `ReactRunner` (`docs/src/modules/components/ReactRunner.tsx`):

1. **Library**: Uses the `react-runner` library (https://github.com/nihgwu/react-runner)
2. **Transpilation**: The library uses Babel Standalone to transpile TSX/JSX code in the browser
3. **Execution**: Transpiled code is executed in a controlled scope with provided imports
4. **Scope Management**: 
   - Provides access to imported modules (MUI components, React, etc.)
   - Handles `process` object for Node.js compatibility
   - Isolates execution context

**How it works**:

```typescript
// User edits code in DemoEditor
const editedCode = `
import Button from '@mui/material/Button';

export default function Demo() {
  return <Button>Click me</Button>;
}
`;

// ReactRunner receives the code
<ReactRunner
  code={editedCode}
  scope={{
    process: {},
    import: {
      '@mui/material/Button': Button,  // Pre-imported components
      'react': React,
      // ... other imports
    }
  }}
  onError={(error) => {/* handle errors */}}
/>

// react-runner library:
// 1. Transpiles TSX/JSX using Babel Standalone
// 2. Executes the transpiled code
// 3. Returns the rendered React element
```

**Technical Details**:

- **TypeScript Support**: TSX code is transpiled to JavaScript using Babel's TypeScript preset
- **No Build Step**: All transpilation happens in the browser at runtime
- **Scope Isolation**: Code runs in a controlled scope to prevent security issues
- **Error Handling**: Syntax and runtime errors are caught and displayed to users
- **Performance**: Transpilation is debounced (300ms) to avoid excessive re-renders

#### Error Display System

When errors occur during code execution, they are displayed visually using the `DemoEditorError` component:

**Error Flow**:

1. **Error Detection**: `react-runner` catches errors during transpilation or execution
2. **Error Callback**: `ReactRunner` calls the `onError` callback with the error message
3. **Debouncing**: Errors are debounced (300ms) to prevent flickering when rapidly typing
4. **Visual Display**: `DemoEditorError` renders the error as a Material-UI Alert component

**Visual Appearance**:

The error is displayed as a red Alert banner positioned at the top center of the code editor:

```tsx
<Alert
  variant="filled"
  severity="error"
  sx={{
    position: 'absolute',
    top: 0,
    left: '50%',
    transform: 'translateX(-50%) translateY(-50%)',
    // Small, compact styling
    fontSize: 12,
    py: '2px',
    px: '6px',
  }}
>
  {errorMessage}
</Alert>
```

**Error Types Handled**:

- **Syntax Errors**: Invalid TSX/JSX syntax, missing brackets, etc.
- **Type Errors**: TypeScript type mismatches (when transpiling TSX)
- **Runtime Errors**: Errors during component execution (undefined variables, etc.)
- **Import Errors**: Missing or incorrect imports

**Error State Management**:

```javascript
// Error state is managed in Demo component
const [debouncedError, setDebouncedError] = React.useState(null);

// ReactRunner passes errors via callback
<ReactRunner
  onError={(error) => {
    // Error is debounced to prevent flickering
    debouncedSetError(error);
  }}
/>

// Error is displayed inside DemoEditor
<DemoEditor>
  <DemoEditorError>{debouncedError}</DemoEditorError>
</DemoEditor>
```

**User Experience**:

- Errors appear immediately when code has issues
- Error messages are clear and actionable
- Errors disappear automatically when code is fixed
- Debouncing prevents error messages from flickering during typing
- Errors are positioned to not obstruct code editing

#### Code Editing Flow

The complete flow from user input to rendered output:

```
1. User types in DemoEditor
   ↓
2. onChange handler updates editorCode state
   ↓
3. Debounced update (300ms) prevents excessive re-renders
   ↓
4. ReactRunner receives updated code
   ↓
5. react-runner transpiles TSX/JSX → JS (using Babel Standalone)
   ↓
6. Transpiled code executes in controlled scope
   ↓
7. React element is returned and rendered
   ↓
8. Errors (if any) are caught and displayed
```

#### Preview Mode vs Live Editing

The system intelligently switches between modes:

- **Preview Mode**: When code panel is closed, shows the `.tsx.preview` snippet
- **Initial State**: When code matches the original, uses pre-rendered bundled component (faster)
- **Live Editing**: When code differs from original, switches to `ReactRunner` for live updates

This is handled in `useDemoElement` hook:

```javascript
// If code matches original and live editing not active → use bundled component
// Otherwise → use ReactRunner for live updates
return editorCode.value === editorCode.initialEditorCode && liveDemoActive === false
  ? BundledComponent
  : LiveComponent;
```

### Code Viewer Tabs

When a demo has relative modules, tabs are shown:

```typescript
// Main file tab
tabs = [
  { module: "./Example.tsx", raw: "..." },
  { module: "./data.ts", raw: "..." },
  { module: "./utils/helper.ts", raw: "..." }
]
```

Each tab shows the full code for that file, and only the main file supports live editing.

## Best Practices

### Creating Demos

1. **File Naming**: Use PascalCase for demo files
2. **Preview Generation**: Keep return JSX simple for preview generation
3. **Relative Imports**: Use relative imports for shared utilities
4. **TypeScript First**: Write demos in TypeScript, JS is auto-generated

### Preview Files

- Previews work best with simple return statements
- Avoid complex logic in the return statement
- Wrapper components (`div`, `Box`, `Stack`) are automatically removed
- Keep previews under 16 lines

### Multiple Files

- Use relative imports for shared code
- Keep file structure flat when possible
- Document complex utilities in comments
- Use TypeScript for better type safety

## Technical Details

### Markdown Processing

The markdown is processed in `packages/markdown/parseMarkdown.mjs`:

```javascript
// Splits markdown into content blocks
const contents = getContents(markdown);
// Returns: [string, {demo: "..."}, string, ...]
```

### Demo Detection

Demos are detected via regex in `packages/markdown/prepareMarkdown.mjs`:

```javascript
if (/^"(demo|component)": "(.*)"/.test(content)) {
  return JSON.parse(`{${content}}`);
}
```

### Relative Module Resolution

The loader (`packages/markdown/loader.mjs`) resolves relative imports:

1. Scans imports in demo files
2. Detects relative paths (`.`, `..`)
3. Resolves file extensions
4. Recursively processes nested imports
5. Groups by variant (JS/TS)

### Preview Extraction

The Babel plugin extracts previews by:

1. Finding the default export function
2. Locating the return statement
3. Extracting JSX from the return
4. Removing wrapper components if needed
5. De-indenting and trimming
